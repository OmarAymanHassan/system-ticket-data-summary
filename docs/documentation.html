Overview and architecture

arch

The task: convert a raw system-ticket text file into per-customer, per-product storytelling summaries generated by an LLM, presented in a Streamlit web app, with a data-quality-safe preprocessing pipeline and a business insights dashboard.

The whole pipeline is a LangGraph graph: every step is a node, and one invoke call carries a file from raw text to finished, verified summaries:

notebooks/ticket_summary.ipynbPrototype and teaching artifact: each step explained, explored line by line with visible output, then assembled into a function
pipeline.pyThe reusable module: all step functions, the output contract, and the compiled LangGraph graph
app.pyStreamlit app: upload, preprocessing report, verified AI summaries, insights dashboard
.envSecrets and model choice (GOOGLE_API_KEY, GEMINI_MODEL); git-ignored, never hard-coded
design rule 01 Validate, never repair silently. Malformed input raises a clear error with line numbers. The one repair that is performed (the column shift, §3.2) is fingerprint-verified and reports every affected ticket number on every run.
design rule 02 One job per function. Each step is a small function used identically by the notebook, the graph and the app - changing a grouping or a prompt touches one place.

Preprocessing: validate and repair

etl

3.1Load and validate

Parses with csv.reader. Accepts files with or without a header line:

  • Line 1 equals the standard 38-column header - header present.
  • Line 1 matches the order-number pattern \d{3}-\d+/\d{2} (e.g. 001-0671177/24) and has 38 fields - headerless file; the standard header is applied and the app reports it.
  • Anything else raises: wrong field count ("different ticket scheme?"), unrecognizable line 1 ("wrong file?"), empty file, or a header with no tickets.
def load_raw_tickets(source):
    ...
    parsed = [r for r in csv.reader(io.StringIO(text)) if r]
    first = parsed[0]
    if first == EXPECTED_COLUMNS:            # line 1 is exactly our header
        header, data, had_header = first, parsed[1:], True
    else:                                    # headerless file
        if len(first) != len(EXPECTED_COLUMNS):
            raise ValueError(f"This file has {len(first)} fields per row, expected 38 ...")
        if not ORDER_NUMBER_PATTERN.fullmatch(first[0]):
            raise ValueError("Line 1 is neither the ticket header nor a ticket row ...")
        header, data, had_header = list(EXPECTED_COLUMNS), parsed, False

    bad = [(line, len(r)) ... if len(r) != n_cols]      # ragged-row check
    if bad:
        raise ValueError(f"Incomplete tickets - ... (line, fields): {bad}")

The schema itself is one verbatim string (EXPECTED_HEADER): a single source of truth copied from the ticket system's export and split into the 38 names.

3.2Repair the shifted rows

The fingerprint: support-team codes (TSCW2, TSCKT, TSCS2) are only ever legal in PLANNING_GROUP_KB. A team code sitting in NOTE_MAXIMUM proves the row slid one column left. The repair is two list operations, applied only to fingerprinted rows:

for r in rows:
    if r[note_idx] in TEAM_CODES:   # team code where the note belongs = shifted row
        r.insert(add_idx, "N/A")    # re-insert the placeholder -> rest slides right
        r.pop()                     # drop the padding beyond the last column
        repaired.append(r[0])
log.info(f"repair: {len(repaired)} shifted tickets fixed: {repaired}")
why this is safe Nothing is invented: every value already exists in the row and only moves back under its correct header. The removed tail is always padding (verified: N/A or empty in all 19 rows). The repair is idempotent - on already-repaired data it finds zero fingerprints. Without it, the LLM would read team codes as customer notes for 19 of 32 tickets.

3.3Convert

The repaired table is persisted as data/tickets_converted.csv and .xlsx - the format conversion the task requires, done after repair so the files carry corrected columns.

3.4Clean

out = out.apply(lambda s: s.str.strip())         # stray spaces
out = out.replace({"N/A": pd.NA, "": pd.NA})     # literal N/A -> real missing values
out["ACCEPTANCE_TIME"] = pd.to_datetime(out["ACCEPTANCE_TIME"], format="%m/%d/%Y %H:%M")
out["COMPLETION_TIME"] = pd.to_datetime(out["COMPLETION_TIME"], format="%m/%d/%Y %H:%M")
out["RESOLUTION_MINUTES"] = (out["COMPLETION_TIME"] - out["ACCEPTANCE_TIME"]).dt.total_seconds() / 60
empty_cols = out.columns[out.isna().all()]       # 100%-empty columns only
out = out.drop(columns=empty_cols)               # drop is logged, never silent

Dates use the default errors="raise": a garbage timestamp stops the run loudly. The empty-column check uses .all(), so a column with even a single real value always survives; after the repair, 10 columns are genuinely empty and dropped.

3.5Filter

Only the predefined categories HDW, NET, KAI, KAV, GIGA, VOD, KAD are kept (isin mask); NTF inquiries and SOP activations are excluded. Result: 32 of 42 tickets, sorted chronologically - order matters for storytelling.

Summary generation and the output contract

llm

Summaries are generated per customer per product - each narrative describes one real customer's history with one product. For the sample file: 3 customers x 6 products = 18 summaries, returned as {customer: {product: {markdown, status, story}}}.

5.1Ticket serialization

The LLM receives each group's tickets as compact chronological lines; only non-missing fields are included - which is why cleaning produced real missing values first:

- 001-0671177/24 | 2024-11-05 11:00 | cat=HDW | issue: Cable Router / WLAN / Slow
  | note: WLAN slow | resolution: WLAN settings optimized / OK

5.2The output contract: schema, validation, evaluation

The summary is not free text - it is a validated contract, built in four layers:

  • Schema - a Pydantic model (five phases, each with ticket_numbers and narrative), enforced natively by the API during generation (response_schema): missing sections or wrong types are impossible. There is deliberately no timeframe field - what the model does not author, it cannot hallucinate.
  • Semantic validators - every ticket reference must match the order-number pattern; "N/A" or padding values are rejected on sight (absence is an empty list, never a value).
  • Reference evaluation - two checklists against the group's real tickets: cited-but-not-real (invented) and real-but-not-cited (omitted). Violations trigger a corrective retry that names the exact mistakes and hands over the true ticket list; persistent failures ship with a visible "unverified" badge, never silently.
  • Computed timeframes - each section's displayed period is looked up from the verified tickets' real timestamps by pandas. A displayed date cannot be hallucinated because the model never writes one.

Phase assignment is order-preserving segmentation: the model walks the chronological tickets and only ever decides "stay in this phase or advance" - never backward. A phase can hold several tickets (one episode, e.g. two calls in one morning) or none. Customer 123's Hardware story, as segmented in a verified run:

guarantee The reference check runs live on every generation - each of the 18 summaries carries a status (verified / corrected / unverified) shown in the app. Validators are inspectors, not editors: a flawed answer is rejected and regenerated by the model, never silently trimmed or patched by our code. Narrative prose remains grounded (the model sees only the ticket lines) and auditable via its per-section citations - verified where verifiable, grounded where not.

5.3Execution and resilience

The structured call is wrapped twice: timeout=90 converts a silently stalled connection into an error instead of an infinite hang, and with_retry(stop_after_attempt=3) converts that error into a fresh attempt - observed in production: a dead connection cost 90s, the retry verified in 6.5s. Every pipeline stage logs a timestamped progress line (summarize [17/18]: -> verified in 4.8s), so a running job always shows where it is. The model name lives in .env only (gemma-4-31b-it for development with its high free quota, gemini-3.5-flash for final runs) - switching models is a one-line change, proven when Google retired gemini-2.5-flash mid-project.

User guide: the Streamlit app

app

7.1Setup (once)

uv sync                            # or: pip install -r requirements.txt
# .env in the project root:
GOOGLE_API_KEY=your-key-here       # https://aistudio.google.com/apikey
GEMINI_MODEL=gemma-4-31b-it

7.2Run

streamlit run app.py               # browser opens at http://localhost:8501

7.3Using the app

  1. Data source: the bundled sample dataset loads automatically; upload a .txt or .csv (with or without the header line) to replace it. The sidebar's "Active data file" panel always shows which file is in use, with a raw download. Invalid files get one clearly worded error instead of a traceback.
  2. Preprocessing report: a data-quality panel lists any repaired shifted tickets; the metric row shows raw vs analyzed counts.
  3. Data tab: the converted table (with CSV download) and the cleaned, product-mapped table.
  4. AI Summaries tab: click Generate summaries, watch per-group progress, then pick a customer and a product to read that story - each summary shows its verification badge (verified / corrected / unverified) and an aggregate reference-check line. Download the complete set as a formatted PDF (one page per customer x product) or as Markdown.
  5. Insights tab: an executive scorecard (repeat rate, first-contact resolution, escalation rate, resolution time, contacts per issue) followed by the business analysis (§08), with product and customer filters - all findings recompute for the current selection. Extra views (most common fixes, FCR by product, team workload, root-cause codes, volume timelines) sit in a clearly labeled expander.

7.4Notebook usage

Open notebooks/ticket_summary.ipynb and Run All: Steps 0-6 run without any API key; Step 7 onward needs GOOGLE_API_KEY in .env. Step 8.5 demonstrates the output contract live (including a deliberately broken story being caught); Step 10 prints the 18 verified summaries; Step 11 reproduces the analysis charts cell by cell.

Measured hold times inside the router chains: 2 to 25 hours per remote fix (best case 6 days, then total device failure). Q3 finding
Every customer is a repeat caller - churn risk, not just cost

All three customers contacted support 10-11 times inside two weeks, and 44% of those contacts were repeats about a product whose previous ticket had already been closed as resolved. First-contact resolution: 100% for four products, 0% for Hardware - one device destroys the KPI. (Broadband also shows 0%, but that is an artifact of the KAI+NET product bundling, not failed fixes.)

Action: treat heavy repeat callers as churn risks - contact them proactively once the device fix ships, before the renewal conversation.
