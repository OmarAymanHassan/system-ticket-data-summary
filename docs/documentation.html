01Overview and architecture

The task: convert a raw system-ticket text file into per-customer, per-product storytelling summaries generated by an LLM, presented in a Streamlit web app, with a data-quality-safe preprocessing pipeline and a business insights dashboard.

The whole pipeline is a LangGraph graph: every step is a node, and one invoke call carries a file from raw text to finished summaries:

load repair convert clean filter map_products summarize
ComponentRole
notebooks/ticket_summary.ipynbPrototype and teaching artifact: each step explained, explored line by line with visible output, then assembled into a function
pipeline.pyThe reusable module: all step functions plus the compiled LangGraph graph
app.pyStreamlit app: upload, preprocessing report, AI summaries, insights dashboard
.envSecrets and model choice (GOOGLE_API_KEY, GEMINI_MODEL); git-ignored, never hard-coded
Design rule 1 Validate, never repair silently. Malformed input raises a clear error with line numbers. The one repair that is performed (the column shift, §3.2) is fingerprint-verified and reports every affected ticket number on every run.
Design rule 2 One job per function. Each step is a small function used identically by the notebook, the graph and the app - changing a grouping or a prompt touches one place.

03Implementation: data preprocessing

3.1 Load and validate

Parses with csv.reader. Accepts files with or without a header line:

  • Line 1 equals the standard 38-column header - header present.
  • Line 1 matches the order-number pattern \d{3}-\d+/\d{2} (e.g. 001-0671177/24) and has 38 fields - headerless file; the standard header is applied and the app reports it.
  • Anything else raises: wrong field count ("different ticket scheme?"), unrecognizable line 1 ("wrong file?"), empty file, or a header with no tickets.
def load_raw_tickets(source):
    ...
    parsed = [r for r in csv.reader(io.StringIO(text)) if r]
    first = parsed[0]
    if first == EXPECTED_COLUMNS:            # line 1 is exactly our header
        header, data, had_header = first, parsed[1:], True
    else:                                    # headerless file
        if len(first) != len(EXPECTED_COLUMNS):
            raise ValueError(f"This file has {len(first)} fields per row, expected 38 ...")
        if not ORDER_NUMBER_PATTERN.fullmatch(first[0]):
            raise ValueError("Line 1 is neither the ticket header nor a ticket row ...")
        header, data, had_header = list(EXPECTED_COLUMNS), parsed, False

    bad = [(line, len(r)) ... if len(r) != n_cols]      # ragged-row check
    if bad:
        raise ValueError(f"Incomplete tickets - ... (line, fields): {bad}")

The schema itself is one verbatim string (EXPECTED_HEADER): a single source of truth copied from the ticket system's export and split into the 38 names.

3.2 Repair the shifted rows

The fingerprint: support-team codes (TSCW2, TSCKT, TSCS2) are only ever legal in PLANNING_GROUP_KB. A team code sitting in NOTE_MAXIMUM proves the row slid one column left. The repair is two list operations, applied only to fingerprinted rows:

for r in rows:
    if r[note_idx] in TEAM_CODES:   # team code where the note belongs = shifted row
        r.insert(add_idx, "N/A")    # re-insert the placeholder -> rest slides right
        r.pop()                     # drop the padding beyond the last column
        repaired.append(r[0])
print(f"Repaired {len(repaired)} shifted tickets: {repaired}")
Why this is safe Nothing is invented: every value already exists in the row and only moves back under its correct header. The removed tail is always padding (verified: N/A or empty in all 19 rows). The repair is idempotent - on already-repaired data it finds zero fingerprints. Without it, the LLM would read team codes as customer notes for 19 of 32 tickets.

3.3 Convert

The repaired table is persisted as data/tickets_converted.csv and .xlsx - the format conversion the task requires, done after repair so the files carry corrected columns.

3.4 Clean

out = out.apply(lambda s: s.str.strip())         # stray spaces
out = out.replace({"N/A": pd.NA, "": pd.NA})     # literal N/A -> real missing values
out["ACCEPTANCE_TIME"] = pd.to_datetime(out["ACCEPTANCE_TIME"], format="%m/%d/%Y %H:%M")
out["COMPLETION_TIME"] = pd.to_datetime(out["COMPLETION_TIME"], format="%m/%d/%Y %H:%M")
out["RESOLUTION_MINUTES"] = (out["COMPLETION_TIME"] - out["ACCEPTANCE_TIME"]).dt.total_seconds() / 60
empty_cols = out.columns[out.isna().all()]       # 100%-empty columns only
out = out.drop(columns=empty_cols)               # drop is printed, never silent

Dates use the default errors="raise": a garbage timestamp stops the run loudly. The empty-column check uses .all(), so a column with even a single real value always survives; after the repair, 10 columns are genuinely empty and dropped.

3.5 Filter

Only the predefined categories HDW, NET, KAI, KAV, GIGA, VOD, KAD are kept (isin mask); NTF inquiries and SOP activations are excluded. Result: 32 of 42 tickets, sorted chronologically - order matters for storytelling.

05Implementation: summary generation

Summaries are generated per customer per product - each narrative describes one real customer's history with one product. For the sample file: 3 customers x 6 products = 18 summaries, returned as {customer: {product: text}}.

5.1 Ticket serialization

The LLM receives each group's tickets as compact chronological lines; only non-missing fields are included - which is why cleaning produced real missing values first:

- 001-0671177/24 | 2024-11-05 11:00 | cat=HDW | issue: Cable Router / WLAN / Slow
  | note: WLAN slow | resolution: WLAN settings optimized / OK

5.2 The prompt contract

The prompt fixes the output structure required by the task: five sections - Initial Issue, Follow-ups, Developments, Later Incidents, Recent Events - each with Timeframe, Ticket Numbers and Narrative. Its rules:

  • Use ONLY the ticket data below; never invent facts.
  • Keep events chronological; split the timeline sensibly across the five sections.
  • If there are few tickets (even a single one), still produce all five sections but keep them very brief.
  • Some tickets are in German - translate everything and write the summary in English.
Audit result An automated audit cross-checked every ticket number mentioned in all 18 generated summaries against each group's actual tickets: zero invented, zero omitted. For one-ticket groups the model fills empty sections with grounded absence statements ("no further tickets recorded") instead of fabricating events - the exact failure-refusal a summarizer should show.

5.3 Execution

The chain STORY_PROMPT | llm | StrOutputParser() is wrapped in .with_retry(stop_after_attempt=3): the Gemini endpoint intermittently drops connections, and one hiccup must not kill an 18-call run. The graph's summarize node loops the groups and reports progress per group. The model name lives in .env only (gemma-4-31b-it for development with its high free quota, gemini-3.5-flash for final runs) - switching models is a one-line change, which proved itself when Google retired gemini-2.5-flash mid-project.

06Code documentation

  • pipeline.py - every function carries a docstring stating its contract; inline comments mark only non-obvious constraints (why utf-8-sig, why the fingerprint is reliable, why .all() not .any(), why retries exist).
  • notebooks/ticket_summary.ipynb - every step is documented three times: a markdown cell explaining the logic in plain words, exploratory cells running each line separately with visible output, then the assembled function. Debug trails (the BOM incident) are kept and labeled for learning value.
  • app.py - section banners per tab; all findings under charts are computed from the loaded data at render time, never hardcoded text.

uv sync                            # or: pip install -r requirements.txt
# .env in the project root:

7.2 Run

streamlit run app.py               # browser opens at http://localhost:8501

7.3 Using the app

  1. Upload the raw ticket .txt in the sidebar - with or without the header line (a banner reports when the standard header was applied automatically).
  2. Preprocessing report: a data-quality panel lists any repaired shifted tickets; the metric row shows raw count, filtered count, products and average resolution time. A clearly worded error appears instead if the file is invalid (wrong scheme, wrong file, empty, broken rows).
  3. Data tab: the converted table (with CSV download) and the cleaned, product-mapped table.
  4. AI Summaries tab: click Generate summaries, watch per-group progress, then pick a customer and a product to read that story. Download all summaries exports the complete set as one Markdown file.
  5. Insights tab: an executive scorecard (repeat rate, first-contact resolution, escalation rate, resolution time, contacts per issue) followed by the business analysis (§08), with product and customer filters - all findings recompute for the current selection. Extra views (most common fixes, FCR by product, team workload, root-cause codes, volume timelines) sit in a clearly labeled expander.

7.4 Notebook usage

Open notebooks/ticket_summary.ipynb and Run All: Steps 0-6 run without any API key; Step 7 onward needs GOOGLE_API_KEY in .env. Step 10 prints the 18 summaries; Step 11 reproduces the insights charts cell by cell.

08Business insights (bonus analysis)

The analysis is structured as the three questions every support dataset must answer. Each chart in the app ends with a finding computed live from the data.

Q1
What breaks the most?
Pareto per device (German names normalized) + tickets per product - where the volume comes from.
Q2
Do fixes hold?
Repeat-contact chains per customer x product, with days between contacts and each chain's final resolution.
Q3
Which customers need attention?
Tickets per customer by product + first vs repeat contacts - the churn-risk view.

Above the questions sits an executive scorecard computed from the filtered data: repeat-contact rate 44%, first-contact resolution 67%, escalation rate 28%, average resolution 33 minutes, 1.8 contacts per issue.

44%of all tickets
Q1 finding
One device is almost half the support volume

14 of 32 analyzed tickets are the cable router - all three customers, the same device, within the same two weeks. No other issue area exceeds 3 tickets.

Action: one targeted fix (firmware push, batch swap, self-service WLAN reset) attacks 44% of the entire queue at once.
100%of repeat chains ended in escalation
Q2 finding
Repeated remote fixes never held

Measured hold times inside the router chains: 2 to 25 hours per remote fix (best case 6 days, then total device failure). Not one repeated remote fix cured the problem - every chain still ended in a device replacement or a technician visit.

Action: pilot an escalation trigger on the second contact for the same issue - and re-measure the cure rate per contact number at production scale before fixing the threshold.
11contacts from one customer in two weeks
Q3 finding
Every customer is a repeat caller

All three customers contacted support 10-11 times inside two weeks, and 44% of those contacts were repeats about a product whose previous ticket had already been closed as resolved. First-contact resolution: 100% for four products, 0% for Hardware - one device destroys the KPI.

Action: treat heavy repeat callers as churn risks, not workload - contact them proactively once the device fix ships, before they consider cancellation.
Scale caveat 42 tickets, 3 customers, 2 weeks. The findings are real in this file and the method is production-ready, but staffing views (weekday x hour heatmap, daily volume, customer timelines) stay illustrative until fed real volume - the app labels them accordingly.
